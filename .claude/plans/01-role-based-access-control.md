## Overview

Currently, all authenticated users have full access to every feature. This plan introduces role-based access control so that:
- **Admin users** see the current admin portal (dashboard, departments, products, transactions, uploads, todos)
- **Regular users** are redirected to a separate user portal (to be defined in a future plan)
- Access is controlled at the database, routing, and UI levels

---

## Phase 1: Database Schema

### 1.1 Create `user_profiles` table

A new table linked to Supabase `auth.users` that stores role and access metadata.

```sql
CREATE TABLE public.user_profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email TEXT NOT NULL,
  username TEXT,
  role TEXT NOT NULL DEFAULT 'user' CHECK (role IN ('admin', 'user')),
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- `id` — references `auth.users(id)`, one profile per auth user
- `role` — either `admin` or `user`
- `is_active` — allows disabling accounts without deleting them

### 1.2 Create `role_permissions` table

Defines which routes/features each role can access.

```sql
CREATE TABLE public.role_permissions (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  role TEXT NOT NULL CHECK (role IN ('admin', 'user')),
  resource TEXT NOT NULL,        -- e.g. 'dashboard', 'products', 'transactions'
  can_view BOOLEAN DEFAULT false,
  can_create BOOLEAN DEFAULT false,
  can_edit BOOLEAN DEFAULT false,
  can_delete BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(role, resource)
);
```

- Each row defines what a role can do with a specific resource
- Resources map to app sections: `dashboard`, `department`, `product`, `transaction`, `upload`, `todo`

### 1.3 Row-Level Security (RLS) Policies

- Add RLS policies to `user_profiles` so users can read their own profile, admins can read/write all
- Add RLS policies to `role_permissions` so only admins can modify permissions
- Update existing table RLS policies to respect user roles

### 1.4 Seed default permissions

Insert default admin permissions (full access to all resources) and default user permissions (limited/no access initially).

### 1.5 Auto-create profile on signup

Create a Supabase database trigger/function that automatically inserts a `user_profiles` row when a new user signs up in `auth.users`, defaulting to `role = 'user'`.

- After fetching the Supabase auth user, also fetch their `user_profiles` row (role, is_active)
- Return a combined user object with role info
- Add a `getUserPermissions()` helper that fetches permissions for the user's role

### 2.2 Create auth middleware or update `(app)/layout.tsx`

- After auth check, fetch user role
- **Admin** (`role = 'admin'`) — render current `AppLayout` with all admin nav items
- **User** (`role = 'user'`) — redirect to `/user/dashboard` (future user portal routes)

### 2.3 Create a `useUserRole` hook

- Client-side hook to fetch current user's profile and permissions
- Cache with React Query
- Used by components to conditionally show/hide UI elements

## Phase 5: Security Hardening

### 5.1 Server-side route protection

- Each route group layout validates the user's role server-side
- Unauthorized access redirects to appropriate portal or shows 403

### 5.2 RLS policy enforcement

- All existing tables (`departments`, `products`, `transactions`, `todos`) get RLS policies that check user role
- Admins: full read/write
- Users: read-only (or scoped access based on permissions)

### 5.3 API-level checks

- All Supabase queries in hooks respect RLS
- Mutation hooks (create/update/delete) are disabled in UI for unauthorized roles
